All network calls are centralised in `src/api/transactions.ts`. Components and hooks never call `fetch` directly — they import the typed functions exported from this module.

## Base URL

The backend base URL is read from the `VITE_API_URL` environment variable at build time:

```typescript src/api/transactions.ts
const API_URL = import.meta.env.VITE_API_URL;
```

Create a `.env` file at the project root to configure it locally:

```bash .env
VITE_API_URL=http://localhost:3000
```

<Warning>
  `VITE_API_URL` must be set before running `vite build`. Omitting it will cause every request to be sent to `undefined/api/…`, resulting in network errors.
</Warning>

## Filters type

Several API functions accept an optional `filters` argument. The shape is defined in `src/types/types.tsx`:

```typescript src/types/types.tsx
export type Filters = {
  type?: string;
  category?: string;
  year?: string;
  month?: string;
}

export type FilterKey = keyof Filters
```

Filters are converted to query string parameters via the native `URLSearchParams` constructor:

```typescript
const params = new URLSearchParams(filters);
const res = await fetch(`${API_URL}/api/transactions?${params}`);
```

## API functions

### Transaction endpoints

<Accordion title="getTransactions">

```typescript src/api/transactions.ts
export async function getTransactions(filters = {}): Promise<Transaction[]> {
  const params = new URLSearchParams(filters);
  const res = await fetch(`${API_URL}/api/transactions?${params}`);
  if (!res.ok) throw new Error(`Error loading transactions: ${res.status}`);
  return res.json();
}
```

Fetches the full list of transactions, optionally filtered by `type`, `category`, `year`, and/or `month`.

</Accordion>

<Accordion title="createTransaction">

```typescript src/api/transactions.ts
export async function createTransaction(
  newItem: Transaction,
): Promise<Transaction> {
  const response = await fetch(`${API_URL}/api/transactions`, {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
    },
    body: JSON.stringify(newItem),
  });

  if (!response.ok) {
    const data = await response.json().catch(() => ({}));
    throw new Error(
      data.message ?? `Error adding transaction (${response.status})`,
    );
  }

  return response.json();
}
```

</Accordion>

## Error handling

Read-only GET requests throw a plain `Error` with the HTTP status code embedded in the message:

```typescript
if (!res.ok) throw new Error(`Error loading transactions: ${res.status}`);
```

Mutating requests (POST / PATCH / PUT / DELETE) attempt to parse the response body first so that server-supplied validation messages are surfaced to the UI:

```typescript
if (!response.ok) {
  const data = await response.json().catch(() => ({}));
  throw new Error(
    data.message ?? `Error adding transaction (${response.status})`,
  );
}
```

The `.catch(() => ({}))` guard ensures the error path does not throw a second exception if the body is not valid JSON.

<Tip>
  Callers (context providers and hooks) catch these errors and store the message in their `error: string | null` state, which the UI can display to the user.
</Tip>
